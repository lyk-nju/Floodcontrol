# Floodcontrol Agent Instructions

本文件适用于 `Floodcontrol/` 目录及其全部子目录。任何在本仓库内工作的 coding agent 都必须遵守。

## 开发日志是完成条件

每一次代码相关修改任务完成后，必须在最终回复用户之前更新：

```text
docs/DEVELOPMENT_LOG.md
```

“代码相关修改”包括：

- Python/CUDA/前端等实现代码；
- 配置、训练或推理脚本；
- 测试、评测与数据处理工具；
- 模型、数据、runtime 或 checkpoint 协议；
- 会约束后续实现的架构设计文档；
- 文件删除、重命名、迁移与兼容层调整。

纯只读分析、代码解释或没有落盘修改的讨论不需要写日志。

## 每条日志必须包含

在 `docs/DEVELOPMENT_LOG.md` 文件末尾追加一个新条目，至少说明：

```text
日期与任务名称
改动类型
实际改动内容
改动理由
验证命令或验证结果
涉及文件
尚未完成的后续事项
```

要求：

- 日志描述最终实际状态，不记录未执行的计划为已完成事项。
- 未运行测试时必须明确记录“未运行”及原因。
- 不得宣称未执行的测试已经通过。
- 不得覆盖、删除或改写既有历史条目。
- 撤销旧改动时追加新的 revert 条目，并引用被撤销的任务。
- 同一次任务涉及多个紧密相关文件时写一个完整条目，不为每个文件制造碎片记录。
- 最终回复用户时简要说明开发日志已经更新。

如果代码修改已经完成但日志尚未更新，则任务仍视为未完成。

## 设计与实现状态必须分开

- 架构文档中的 `LOCKED` 表示协议已确认，不表示代码已经落地。
- 日志只能记录实际写入仓库的变化，并准确区分“文档协议变更”和“代码实现变更”。
- 不得用开发日志替代详细设计文档；设计内容仍写入 `docs/rearchitecture/` 对应主题。


## 代码开发规范
- 在你每次要写入代码前，应该先审视一下现有代码框架及代码结构，同时把任务进行一个消化，看看如何在已有框架内用最小的代码成本实现，让整个框架清晰可维护，而不是直接完成代码工作，要把边界明确、框架可维护作为第一要务。
